Encodage et caractère spéciaux

Éviter les erreurs d'interprétation

La question de l'encodage et de la gestion des caractères spéciaux est particulièrement prégnante en Français, puisque la langue utilise des apostrophes et de l'accentuation.

Deux exemples de problèmes communément rencontrés lors de l'automatisation de scripts Gherkin

L'utilisation des apostrophes

Le texte se trouvant entre deux apostrophes sera considéré par Cucumber comme une donnée de test (chaîne de caractères)

Zoom

Si le testeur n'échappe pas ces apostrophes, alors voici ce que comprendra Cucumber :

@When("Je valide l{string}un article dont le prix est {int} euros")
public void je_valide_l_un_article_dont_le_prix_est_euros(String string, Integer int1) {
    // Write code here that turns the phrase above into concrete actions
}

Gérer l'encodage du serveur d'exécution

En français, les mots-clés Gherkin comprennent des accents qu'un éditeur de texte Gherkin interprétera correctement si le script est précédé de l'instruction #language :fr.

Zoom

Néanmoins, il se peut que le serveur d'exécution automatisé (sur lequel repose l'interpréteur Cucumber), utilise un encodage standard qui ne lit pas les accent ( ex : Cp1252).

Si tel est le cas, l'exécution renverrait un message d'erreur comme suit :

Zoom

Pour résoudre ce problème, assurez-vous que l'encodage du serveur d'exécution utilise un type d'encodage adéquat (ex :UTF-8)
