Capitaliser sur un patrimoine existant

Le workflow d'automatisation des tests Gherkin présuppose, à raison, que l'écriture des scripts par les testeurs fonctionnels précède l'implémentation par les automaticiens.

À ce principe de base, il s'agit néanmoins d'apporter une nuance, puisqu'après avoir implémenté plusieurs tests, l'équipe dispose d'un patrimoine qu'elle peut réutiliser et valoriser.

Ainsi, lors de l'écriture de nouveaux scénarios, le testeur fonctionnel pourra piocher dans cette bibliothèque de mots-clés implémentés.

Une équipe agile développe un site de e-commerce.

Sprint 2

Passé en succès à l'issue du premier sprint, ce test devient candidat à l'automatisation. Voici les méthodes implémentées par l'automaticien au cours du 2ème sprint.

public class AnnotationSteps {
	
	Utilisateur utilisateur;
	Produit produit;
	
	@Given("Mon solde est de {int} euros")
	public void mon_solde_est_de_euros(Integer solde) {
		utilisateur  = new Utilisateur();
		utilisateur.setSolde(solde);
	}
	@When("Je valide l\\'achat d\\'un article dont le prix est {int} euros")
	public void je_valide_l_achat_d_un_article_dont_le_prix_est_euros(Integer prix) {
		produit.setPrice(prix);
		int newSolde = utilisateur.acheteProduit(produit);
		utilisateur.setSolde(newSolde);
	}
	@Then("Mon solde est désormais de {int} euros")
	public void mon_solde_est_désormais_de_euros(int expectedSolde) {
	    assertEquals(expectedSolde, utilisateur.getSolde());
	}
Sprint 3
Au cours du 3ème sprint, le testeur doit valider la fonctionnalité permettant à l'utilisateur de créditer son compte.
Une manière intelligente de faire est :
  • de procéder à l'inventaire des mots-clés déjà implémentés pour réutilisation
  • d'identifier les instructions spécifiques au nouveau scénario
  • de communiquer ses choix à l'automaticien
