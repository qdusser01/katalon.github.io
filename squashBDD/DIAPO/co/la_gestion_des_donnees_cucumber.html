Données de test et paramètres de méthode

Autre source d'erreur ou d'incompréhension entre testeurs et automaticiens au début d'un projet BDD : la gestion des données de test par le framework d'automatisation.

Pour éviter ces désagréments, une compréhension de la syntaxe Gherkin et de son interprétation par Cucumber est primordiale.

Données de test et paramètres de méthode

Les données de test sont interprétées comme des paramètres de méthode.

  • Les chaînes de caractères sont des paramètres de type 'String'

  • Les données numériques sont des paramètres de type entier ('int', 'byte', 'short', 'long')

  • Les tables de données sont des tables, des listes, des listes de listes voire des objets comportant des attributs

Chaînes de caractères

Le scénario ci-dessous comporte deux données de test sous forme de chaînes de caractères : il s'agit de la valeur du login et du password utilisé pour vérifier la fonctionnalité de connexion d'un utilisateur enregistré.

Zoom

Le test étant un test d'application web, l'automaticien utilisera le framework d’automatisation Selenium (en java).

En regardant de près le code ci-dessous, on s'aperçoit que la deuxième méthode (@Quand...) comporte deux paramètres de type String. Une valeur sera affectées à ces paramètres lors de l'appelle de cette méthode par le script Gherkin lorsque celui-ci sera exécuté... au cours d'une campagne d'exécution automatisée.

public class AnnotationSteps {
	
	WebDriver driver;
	@Etantdonnéque("L'utilisateur est sur la page de connexion")
	public void l_utilisateur_est_sur_la_page_de_connexion() {
		System.setProperty("webdriver.gecko.driver", "src/test/resources/squash/geckodriver.exe");
		driver = new FirefoxDriver();
		driver.get("http://un_site.com");
	}
	@Quand("l'utilisateur se connecte avec le login {string} et le password {string}")
	public void l_utilisateur_se_connecte_avec_le_login_et_le_password(String string1, String string2) {
		driver.findElement(By.id("username")).clear();
		driver.findElement(By.id("username")).sendKeys(string1);
		driver.findElement(By.id("password")).clear();
		driver.findElement(By.id("password")).sendKeys(string2);
	}
	@Alors("La page d'accueil est affichée")
	public void la_page_d_accueil_est_affichée() {
		assertEquals("Page d'accueil", driver.getTitle());
		driver.quit();
	}
	
}
Données numériques
L'exemple ci-dessous présente l'implémentation d'un scénario d'addition d'une calculatrice. Les données numérique deviennent des paramètres de type entier.
Zoom
public class CalculatorRunSteps {
 
    private int total;
 
    private Calculator calculator;
 
    @Before
    private void init() {
        total = -999;
    }
 
    @Given("^I have a calculator$")
    public void initializeCalculator() throws Throwable {
        calculator = new Calculator();
    }
 
    @When("^I add (-?\\d+) and (-?\\d+)$")
    public void testAdd(int num1, int num2) throws Throwable {
        total = calculator.add(num1, num2);
    }
 
    @Then("^the result should be (-?\\d+)$")
    public void validateResult(int result) throws Throwable {
        Assert.assertThat(total, Matchers.equalTo(result));
    }
}
Tables de données
L'implémentation d'une table de données est un peu plus complexe dans la mesure où l'automaticien choisira une manière de manipuler ces données selon deux facteurs relativement subjectifs :
  • le nombre de colonnes de la table
  • son appétence personnelle à la manipulation de données sous forme de liste, de table, d'objet, de map etc...
On prendra l'exemple d'une table de donnée à deux colonnes (couples clés-valeur)
Zoom
Ci-dessous, l'automaticien a choisi de traiter les données du tableau sous la forme d'une collection de type Map appelé listePrix qui enregistre des couples clé-valeur.
La méthode générique annotée @Quand multiplie le prix du produit par son nombre d’occurrence (ex. : 2 * le prix du donut) et ajoute le résultat à la somme totale. De cette manière elle peut être appelée plusieurs fois au sein d'un même scénario (comme c'est le cas dans le script Gherkin ci-dessus).
public class AnnotationSteps {
	
	
	private Map<String, Integer> listePrix;
	private int totalSum;
	 
	@Soit("le prix de produit de cette liste")
	public void le_prix_de_produit_de_cette_liste(Map<String, Integer> listePrix) {
		 this.listePrix = listePrix;
	}
	@Quand("je commande {int} {string}")
	public void je_commande(int nombreDeProduit, String produit) {
		int firstPrice = listePrix.get(produit);
        totalSum += firstPrice * nombreDeProduit;
	}
	@Alors("je dois payer {int} euros")
	public void je_dois_payer_euros(int prixAttendu) {
		assertEquals(prixAttendu, totalSum);
	}
	
}
Ce dernier exemple illustre parfaitement le fait que l'implémentation d'une méthode générique (avec données de test en paramètres) permet de factoriser le code.
Les méthodes implémentées constituent à terme un patrimoine réutilisable à loisir par les testeurs fonctionnels pour écrire de nouveau script Gherkin.
