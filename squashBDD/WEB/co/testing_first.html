Testing first - les méthodes de développement Agile

En contexte agile, l'activité de test est naturellement présente à tous les niveaux : unitaire, intégration, système, acceptation. Néanmoins, la logique adoptée diffère des cycles de développement traditionnel. On parle ici de Testing first

Tester le code

Motivées par l'adage désormais connu :« plus une anomalie est découverte tard, plus sa correction coûte cher », les méthodes agiles tendent à responsabiliser les développeurs sur la qualité de leur code.

Tester au plus, donc au niveau unitaire, c'est le but de certaines pratiques affichées par les méthodes agiles, notamment par XP Extrême Programming, pour laquelle « tout code doit être testé unitairement ». Les tests Unitaires sont écrits par les développeurs pour tester les composants de façon unitaire. Ils s'inscrivent en général dans une stratégie de développement dirigé par les tests (TDD).

Test Driven Developement (TDD)

Le principe du TDD va à contre-courant de la logique de test traditionnelle. L'idée n'est pas de tester un code déjà implémenté, mais de partir de l'écriture du test (méthode de test unitaire), pour ensuite écrire le code qui permettra au test de passer en succès.

Pair programming

XP promeut le Pair Programming. La programmation se fait à deux. L'un est le driver. C'est lui qui tient le clavier. Le second, appelé partner (ou copilote) est là pour l'aider en suggérant de nouvelles possibilités ou en décelant d'éventuels problèmes. Les développeurs changent fréquemment de partenaire ce qui permet d'améliorer la connaissance collective de l'application et d'améliorer la communication au sein de l'équipe.

Intégration continue

Chaque modification du code source donne lieu à une vérification automatisée de sa qualité. On rencontre sa mise en œuvre sous deux formes :

  • L'intégration continue dans sa forme simple, qui consiste à surveiller les modifications dans le code : dès qu'un changement est détecté, le serveur d'exécution compile et teste le code (unitaire)

  • L'intégration continue dans sa forme complexe, qui consiste à suivre la santé d'un projet : Le serveur d'exécution opère régulièrement des analyses qualimétriques et fournit des métriques sur la qualité du développement. Il compile, teste le code et déploie l'application. Passe les tests système (TNR, performance, stress, sécurité...) et créé un reporting public (client, développeurs, MOA & AMOA).

Tester les fonctionnalités

Behaviour Driven Developement (BDD)

Comme le TDD, le Behaviour Driven Developpement (BDD) formalise la logique d'écriture des tests avant l'implémentation du code, mais au niveau système. Voici ce qui distingue le BDD du TDD.

  • Le BDD se concentre sur l'aspect comportemental attendu du système et non sur les fonctions du code. Il garantit la qualité fonctionnelle là où le TDD garantit la qualité technique du code.

  • Le BDD donne une meilleure compréhension de ce que le système devrait faire du point de vue du développeur ET du client, puisqu'il utilise un langage naturel plutôt qu'informatique pour les scenarii.

  • Le BDD permet au développeur et au client de travailler en commun sur l'analyse des exigences contenues dans le code source.

Le besoin fonctionnel guide le développement de l'application

Acceptance Test Driven Developement (ATDD)

Basé sur les mêmes concepts que le TDD, l'ATDD adopte le point de vue de l'utilisateur final.
